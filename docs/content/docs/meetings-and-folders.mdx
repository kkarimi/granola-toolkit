## Folders

`folder` makes Granola folders a first-class concept in the toolkit.

The current surface includes:

- `granola folder list`
- `granola folder view <id|name>`
- `granola meeting list --folder <id|name>`
- `granola notes --folder <id|name>`
- `granola transcripts --folder <id|name>`

Examples:

```bash
granola folder list
granola folder view Team
granola meeting list --folder Team
```

## Meetings

`meeting` combines API-backed notes with the local transcript cache so you can inspect one meeting
at a time.

Once you have run `granola sync`, the toolkit also keeps a local full-text search dataset for
titles, notes, transcripts, folders, tags, and generated artefacts.

Examples:

```bash
granola search planning decision
granola meeting list --limit 10
granola meeting list --search planning
granola meeting view 1234abcd
granola meeting notes 1234abcd
granola meeting transcript 1234abcd --format json
granola meeting export 1234abcd --format yaml
granola meeting open 1234abcd
```

## Browser and TUI Navigation

Folders also drive interactive navigation:

- the browser workspace has a dedicated folder pane plus an All meetings scope
- the browser search box uses the shared local search dataset when it is available
- the TUI keeps folders inside the navigation flow instead of treating them as an afterthought
- TUI quick open falls back to the local search dataset when a title or id lookup is not enough
- generated notes/enrichment artefacts feed back into the same search index, so one query can find
  a meeting by its transcript, its original notes, or the AI output you approved later
- URL state and attach flows preserve folder and meeting selection

## Why This Matters

Most real users do not want one undifferentiated list of every meeting they have ever recorded.
Treating folders as first-class keeps exports, scripts, and interactive browsing aligned with how
Granola is actually organised.
