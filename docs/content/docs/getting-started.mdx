## Install

```bash
npm install -g granola-toolkit
granola --help
```

If you do not want a global install:

```bash
npx granola-toolkit --help
```

The package exposes both `granola` and `granola-toolkit` as executable names.

If you prefer not to use npm at all, each GitHub release also ships standalone archives for macOS
arm64, Linux x64, and Windows x64. Extract the archive and run `granola` (or `granola.exe` on
Windows).

If you use external agents, the repo also ships an OpenClaw-ready skill at
`skills/openclaw-granola-toolkit/SKILL.md`.

## First Run

Create a local project bootstrap first:

```bash
granola init --provider openrouter
```

That writes:

- `.granola.toml`
- `./.granola/agent-harnesses.json`
- `./.granola/automation-rules.json`
- starter prompt files under `./.granola/prompts/`

The generated files are intentionally local and editable. They give you a working starting point
for team notes and customer follow-up notes without forcing you to reverse-engineer the JSON shape
first.

Store a Granola Personal API key once:

```bash
granola auth login --api-key grn_...
granola auth status
```

If you chose `--provider openrouter` or `--provider openai`, export the matching provider key in
the shell or service that runs your sync loop:

```bash
export OPENROUTER_API_KEY=...
# or
export OPENAI_API_KEY=...
```

If you prefer to reuse the desktop app session, `granola auth login` still imports it from
`supabase.json`.

Then verify that folders and meetings resolve cleanly:

```bash
granola service start --config ./.granola.toml
granola automation rules
granola automation runs
granola search customer onboarding
granola folder list
granola meeting list --limit 10
granola meeting view 1234abcd
```

If you prefer to start in the browser instead of the terminal, run:

```bash
granola web
```

The first-run flow in the web workspace will guide you through the same setup in order: store a
Granola API key, run the first meeting import, choose an agent provider, and create a starter
reviewable notes pipeline before it drops you into the full workspace.

`granola web` now prefers the reusable background-service path, so for many users the command above
is enough to both launch the browser and warm the local sync runtime.

If you want to warm the service before opening the browser, or keep it running headlessly, use:

```bash
granola service start
granola web
granola attach
granola service status
```

## Typical Starting Commands

```bash
granola service start
granola sync
granola search onboarding risk
granola notes
granola notes --folder Team
granola transcripts
granola web
granola attach
```

## When to Use Which Surface

- Use the CLI when you want scripts, exports, or stable machine-readable output.
- Use `granola service start` when you want the sync loop and local workspace runtime to stay warm.
- Use `granola web` when you want quick navigation, tabs, and export actions in a browser.
- Use `granola attach` when you want the same shared runtime in a keyboard-first terminal UI.
