## Install

```bash
npm install -g @kkarimi/gran
gran --help
```

If you do not want a global install:

```bash
npx @kkarimi/gran --help
```

The package exposes `gran` as the executable name.

If you prefer not to use npm at all, each GitHub release also ships standalone archives for macOS
arm64, Linux x64, and Windows x64. Extract the archive and run `gran` (or `gran.exe` on
Windows).

If you use external agents, the repo also ships an OpenClaw-ready skill at
`skills/openclaw-gran/SKILL.md`.

## First Run

Create a local project bootstrap first:

```bash
gran init
```

That writes:

- `.gran.json`
- `./.gran/agent-harnesses.json`
- `./.gran/automation-rules.json`
- starter prompt files under `./.gran/prompts/`

The generated files are intentionally local and editable. They give you a project-local home for
config, targets, prompts, and future automation without forcing you to reverse-engineer the JSON
shape first.

The smoothest first run today is browser-first:

```bash
gran web
```

The first-run flow in the web workspace will guide you through the two things you actually need
before the app becomes useful:

1. save a Granola API key
2. run the first meeting import

After that, you land in the workspace with your local archive ready to browse.

`gran web` prefers the reusable background-service path, so for many users the command above is
enough to both launch the browser and warm the local sync runtime.

If you prefer CLI auth instead, store a Granola Personal API key once:

```bash
gran auth login --api-key grn_...
gran auth status
```

If you prefer to reuse the desktop app session, `gran auth login` imports it from
`supabase.json`.

Then run the first sync and verify that folders and meetings resolve cleanly:

```bash
gran service start
gran sync
gran folder list
gran meeting list --limit 10
gran search onboarding
```

## Choose A Surface

### Browser Workspace

```bash
gran web
```

Use this when you want the calmest first-run path and the easiest place to inspect sync state,
folders, review queues, targets, and diagnostics.

### Terminal Workspace

```bash
gran service start
gran tui
```

Use this when you want a keyboard-first workspace on the same shared local runtime.

### Headless Local Runtime

```bash
gran service start
gran attach
gran service status
```

Use this when you want the sync loop and local runtime running before you open either browser or
terminal UI.

## Common Next Steps

- `gran export` when you want notes and transcripts in durable local files
- `gran targets add ...` when you want a named destination like an Obsidian vault
- `Settings -> Automation` when you want review-first AI notes
- `gran attach` when you want to reuse the running service in the terminal

For concrete examples, see [`Workflows`](/docs/workflows).
