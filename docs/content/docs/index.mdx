Gran 👵🏻 is an unofficial open-source toolkit for getting more out of Granola than the
desktop app and official exports expose on their own.

It gives you a local-first workbench for your Granola archive:

- sync meetings, notes, transcripts, folders, and change history into a local runtime
- browse that state from the CLI, browser workspace, and terminal UI
- export notes and transcripts without getting trapped in a closed format
- reuse the same local-first core from a Node and TypeScript SDK
- run your own agents and reviewable pipelines against meeting transcripts
- script and extend the whole thing in plain TypeScript and local files

The fastest way to start is `gran init`. In an interactive terminal, it now guides you through
connection, first import, and opening either the browser workspace or the terminal workspace before
you need to think about automation or PKM targets.

It has three user-facing surfaces on one shared TypeScript core:

- batch CLI commands
- a browser workspace from `gran web`
- a terminal workspace from `gran tui`

## What It Is Good At

- keeping a local index, local snapshot, and sync loop around your Granola archive
- opening the same meetings from scripts, the browser, and the terminal
- exporting notes and transcripts into durable local files
- running BYOA prompts and reviewable note pipelines
- giving you an open-source seam when Granola itself is missing a feature you need

## Core Ideas

### Local-first sync and ownership

Gran 👵🏻 is built around the idea that your meeting archive should be usable from a local
runtime you control, with an explicit sync loop, a local index, local snapshots, search, exports,
and background work.

### Bring your own agents

Transcripts and notes are not the end of the flow. You can layer local harnesses, `AGENT.md`-style
prompts, review queues, and downstream actions on top of them.

### One shared core across surfaces

CLI commands, the browser workspace, and the TUI all talk to the same application core for auth,
meeting discovery, folders, transcripts, notes, automation, and export jobs.

### Open and hackable by design

Registries, local config files, prompt files, and documented service seams are there so you can
extend the toolkit instead of waiting for an official integration to exist.
