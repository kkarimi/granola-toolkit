## Rule Files

The toolkit reads automation rules from a JSON file. By default it uses the shared toolkit data
directory:

- macOS: `~/Library/Application Support/granola-toolkit/automation-rules.json`
- Linux: `~/.config/granola-toolkit/automation-rules.json`

Override that path with either:

- `--rules /path/to/rules.json`
- `GRANOLA_AUTOMATION_RULES_FILE`
- `automation-rules-file = "/path/to/rules.json"` in `.granola.toml`

## Rule Shape

```json
{
  "rules": [
    {
      "id": "team-transcript",
      "name": "Team transcript ready",
      "when": {
        "eventKinds": ["transcript.ready"],
        "folderNames": ["Team"],
        "tags": ["customer"],
        "transcriptLoaded": true
      },
      "actions": [
        {
          "id": "notes-markdown",
          "kind": "export-notes",
          "outputDir": "./automation/notes",
          "scopedOutput": true
        },
        {
          "id": "summarise",
          "kind": "command",
          "command": "/bin/sh",
          "args": ["-lc", "cat >/tmp/granola-event.json"]
        },
        {
          "id": "review",
          "kind": "ask-user",
          "prompt": "Review this transcript before sharing it"
        }
      ]
    }
  ]
}
```

Each rule can contain zero or more actions. Current action kinds are:

- `export-notes`
- `export-transcript`
- `command`
- `ask-user`

`command` actions receive a JSON payload on stdin by default, including the matched event, rule,
and meeting bundle when the meeting still exists.

`ask-user` actions do not perform side effects immediately. They create a pending automation run
that can be approved or rejected later from the CLI, web workspace, or TUI.

The toolkit stores:

- raw sync events in `sync-events.jsonl`
- rule matches in `automation-matches.jsonl`
- action runs in `automation-runs.jsonl`

All three live in the shared toolkit data directory so watch loops, `serve`, `web`, and `tui` see
the same history.

Available match fields:

- `eventKinds`
- `folderIds`
- `folderNames`
- `meetingIds`
- `tags`
- `titleIncludes`
- `titleMatches`
- `transcriptLoaded`

Fields are combined with `AND`, while multiple values inside one field are treated as `OR`.

## Commands

Inspect the configured rules:

```bash
granola automation rules
granola automation rules --format json
```

Inspect recent rule matches produced by sync runs:

```bash
granola automation matches
granola automation matches --limit 50
```

Inspect recent action runs and resolve pending approval items:

```bash
granola automation runs
granola automation runs --status pending
granola automation approve <run-id>
granola automation reject <run-id> --note "not relevant"
```

## How It Works

`granola sync` and `granola sync --watch` enrich each durable sync event with the meeting title,
folders, tags, and transcript readiness. The toolkit then flows through three layers:

- sync discovers change and persists events
- rules decide whether a change is interesting
- actions execute once per matched event/action pair and persist their final run state

That idempotent run log is what keeps automation observable instead of burying side effects inside
the sync loop.

## Surface Support

- CLI: inspect rules, matches, and runs, and resolve pending `ask-user` runs
- Web: browse recent automation runs and approve or reject pending items
- TUI: open the automation overlay with `u` and resolve pending items from the terminal

## Notes on Exports

Automation exports run through the same file writers and export-job history as manual exports, but
they do not hijack the active UI view. Meeting-scoped automation exports default to stable
subdirectories under `_meetings/<meeting-id>/` when `scopedOutput` is enabled.
